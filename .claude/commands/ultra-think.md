# 深い分析と問題解決モード

深い分析と問題解決モード

## 手順

1. **ウルトラシンクモードの初期化**
   - 強化された分析的思考のリクエストを認識する
   - 深く体系的な推論のためのコンテキストを設定する
   - 問題空間を包括的に探索する準備をする

2. **問題または質問の解析**
   - $ARGUMENTS から核心的な課題を抽出する
   - すべてのステークホルダーと制約を特定する
   - 暗黙の要件と隠れた複雑さを認識する
   - 前提を疑い、未知の要素を明らかにする

3. **多次元分析**
   問題を複数の角度からアプローチする:

   ### 技術的視点

   - 技術的な実現可能性と制約を分析する
   - スケーラビリティ、パフォーマンス、保守性を考慮する
   - セキュリティへの影響を評価する
   - 技術的負債と将来への対応を評価する

   ### ビジネス的視点

   - ビジネス価値と ROI を理解する
   - 市場投入までの時間的プレッシャーを考慮する
   - 競争優位性を評価する
   - リスク対リターンのトレードオフを評価する

   ### ユーザー視点

   - ユーザーのニーズとペインポイントを分析する
   - ユーザビリティとアクセシビリティを考慮する
   - ユーザーエクスペリエンスへの影響を評価する
   - エッジケースとユーザージャーニーを考慮する

   ### システム視点

   - システム全体への影響を考慮する
   - 統合ポイントを分析する
   - 依存関係と結合度を評価する
   - 創発的な動作を考慮する

4. **複数の解決策を生成**
   - 少なくとも 3〜5 つの異なるアプローチをブレインストーミングする
   - 各アプローチについて以下を考慮する:
     - 長所と短所
     - 実装の複雑さ
     - 必要なリソース
     - 潜在的なリスク
     - 長期的な影響
   - 従来型と創造的な解決策の両方を含める
   - ハイブリッドアプローチを検討する

5. **深掘り分析**
   最も有望な解決策について:
   - 詳細な実装計画を作成する
   - 潜在的な落とし穴と緩和戦略を特定する
   - 段階的アプローチと MVP を検討する
   - 二次的・三次的影響を分析する
   - 障害モードと復旧について検討する

6. **分野横断的思考**
   - 他の業界や分野からの類似点を引き出す
   - 異なるコンテキストからのデザインパターンを適用する
   - 生物学的または自然システムのアナロジーを考慮する
   - 既存の解決策の革新的な組み合わせを探す

7. **検証と改良**
   - 各解決策に対して悪魔の代弁者（Devil's Advocate）を演じる
   - 弱点と盲点を特定する
   - 「もし〜だったら」シナリオを検討する
   - 前提をストレステストする
   - 意図しない結果を探す

8. **洞察の統合**
   - すべての視点からの洞察を統合する
   - 重要な意思決定要因を特定する
   - 重要なトレードオフを強調する
   - 革新的な発見を要約する
   - 問題空間のニュアンスのある見方を提示する

9. **構造化された推奨事項の提供**
   調査結果を明確な構造で提示する:

   ```
   ## 問題分析
   - 核心的な課題
   - 主要な制約
   - 成功の重要要因

   ## 解決策オプション
   ### オプション 1: [名前]
   - 説明
   - 長所/短所
   - 実装アプローチ
   - リスク評価

   ### オプション 2: [名前]
   [同様の構造]

   ## 推奨事項
   - 推奨アプローチ
   - 根拠
   - 実装ロードマップ
   - 成功指標
   - リスク緩和計画

   ## 代替的視点
   - 反対意見
   - 将来の考慮事項
   - さらなる調査が必要な領域
   ```

10. **メタ分析**
    - 思考プロセス自体を振り返る
    - 不確実な領域を特定する
    - バイアスや限界を認識する
    - 必要な追加の専門知識を提案する
    - 推奨事項に対する信頼度レベルを提供する

## 使用例

```bash
# アーキテクチャの意思決定
/ultra-think マイクロサービスに移行すべきか、モノリスを改善すべきか？

# 複雑な問題解決
/ultra-think コストを削減しながら 10 倍のトラフィックに対応するにはどうすればよいか？

# 戦略的計画
/ultra-think 次世代プラットフォームにどの技術スタックを選択すべきか？

# 設計上の課題
/ultra-think 後方互換性を維持しながら、API をより開発者フレンドリーにするにはどうすればよいか？
```

## 主要原則

- **第一原理思考（First Principles Thinking）**: 基本的な真実にまで分解する
- **システム思考（Systems Thinking）**: 相互接続とフィードバックループを考慮する
- **確率的思考（Probabilistic Thinking）**: 不確実性と範囲を扱う
- **逆転の発想（Inversion）**: 何をすべきかだけでなく、何を避けるべきかを考える
- **二次的思考（Second-Order Thinking）**: 結果の結果を考慮する

## 出力の期待値

- 包括的な分析（通常 2〜4 ページの洞察）
- トレードオフを伴う複数の実行可能な解決策
- 明確な推論の連鎖
- 不確実性の認識
- 実行可能な推奨事項
- 新しい洞察や視点
